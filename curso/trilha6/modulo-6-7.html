üìê Principios dos Prompts GIPM

No GIPM, prompts vagos geram respostas vagas. Um bom prompt e especifico, fornece contexto, define formato de saida e inclui restricoes claras. A IA e uma ferramenta - quanto melhor a instrucao, melhor o resultado.

Anatomia de um Prompt GIPM

1.
Papel (Role) - Quem a IA deve ser: "Voce e um arquiteto de software senior..."
2.
Contexto - Background do projeto, stack, restricoes
3.
Tarefa - O que precisa ser feito, especificamente
4.
Formato - Como a resposta deve ser estruturada
5.
Restricoes - O que NAO fazer, limites, requisitos nao-funcionais

‚ùå Prompt Ruim

"Crie uma API de login"

‚úÖ Prompt Bom

"Voce e um dev backend. Crie endpoint POST /auth/login em Node.js + Express que valida email/senha, retorna JWT. Siga o padrao do projeto em src/. Inclua rate limiting."

üîç Prompts de Analise

Use esses prompts na fase de descoberta e definicao. Eles ajudam a criar o Project Brief, refinar requisitos e analisar stakeholders. A IA atua como um analista de negocios experiente.

Prompt: Criar Project Brief

# ROLE
Voce e um analista de negocios senior especializado em projetos de software.

# CONTEXTO
Estou iniciando um novo projeto: [DESCRICAO DO PROJETO]

# TAREFA
Crie um Project Brief completo seguindo o template GIPM incluindo:
- Visao e objetivo
- Problema sendo resolvido
- Stakeholders
- Restricoes e premissas
- Metricas de sucesso

# FORMATO
Markdown com secoes claras e bullet points.
Uso: Inicio do projeto, apos reuniao inicial com stakeholders

üí° Outros Prompts de Analise

  • ‚Ä¢ Refinar Requisitos - Transforma ideias vagas em requisitos especificos
  • ‚Ä¢ Analise de Stakeholders - Mapeia interesses e poder de cada stakeholder
  • ‚Ä¢ Analise de Riscos - Identifica riscos e mitigation strategies

üèóÔ∏è Prompts de Arquitetura

Use quando estiver definindo como construir. A IA pode propor arquiteturas, revisar decisoes existentes, e ajudar no design de APIs. Sempre forneca o contexto tecnico do projeto.

Prompt: Propor Arquitetura

# ROLE
Voce e um arquiteto de software senior com experiencia em sistemas distribuidos.

# CONTEXTO
Projeto: [NOME]
Requisitos: [PRD RESUMIDO]
Restricoes: [TIME, BUDGET, STACK]

# TAREFA
Proponha uma arquitetura incluindo:
- Tech stack com justificativas
- Diagrama de componentes (ASCII ou Mermaid)
- Padroes arquiteturais escolhidos
- Trade-offs e riscos

# RESTRICOES
- Time de 3 devs
- MVP em 2 meses
- Budget limitado (sem enterprise licenses)

Design de API

Cria especificacao OpenAPI, define endpoints, metodos, e schemas de request/response.

Revisar Arquitetura

Analisa arquitetura existente, identifica problemas e sugere melhorias.

üíª Prompts de Desenvolvimento

Use quando estiver codificando. Esses prompts ajudam a implementar features, refatorar codigo, e fazer debug. Sempre inclua o contexto do codigo existente.

Prompt: Implementar Feature

# ROLE
Voce e um desenvolvedor fullstack experiente em [STACK].

# CONTEXTO
- Stack: Next.js 14, TypeScript, Prisma, PostgreSQL
- Estrutura: src/app para routes, src/lib para utils
- Codigo existente: [COLE ARQUIVOS RELEVANTES]

# TAREFA
Implemente a Story: [COLE A STORY COMPLETA]

# RESTRICOES
- Siga os padroes do codigo existente
- Inclua tratamento de erros
- Nao use bibliotecas nao listadas no package.json
- Codigo em TypeScript strict

Prompt: Debug

Contexto: Forneca o erro completo, codigo relevante, e o que voce ja tentou.

Tarefa: "Identifique a causa raiz do erro e proponha correcao com explicacao."

Prompt: Refatorar

Contexto: Codigo atual, problemas identificados, objetivo da refatoracao.

Tarefa: "Refatore para [OBJETIVO]. Mantenha comportamento. Explique mudancas."

üß™ Prompts de Testes

Use para garantir qualidade. A IA pode gerar testes unitarios, criar planos de teste, e identificar edge cases que voce pode ter esquecido.

Prompt: Gerar Testes Unitarios

# ROLE
Voce e um QA engineer especializado em testes automatizados.

# CONTEXTO
- Framework de testes: Jest + React Testing Library
- Codigo a testar: [COLE O CODIGO]

# TAREFA
Crie testes unitarios cobrindo:
- Happy path
- Inputs invalidos
- Edge cases
- Erros esperados

# FORMATO
Arquivo de teste completo, pronto para executar.

Plano de Testes

Cria plano de testes completo para uma feature, incluindo casos manuais e automatizaveis.

Identificar Edge Cases

Analisa requisitos e identifica cenarios de borda que devem ser testados.

üëÄ Prompts de Code Review

Use para validar codigo antes do merge. A IA pode revisar PRs, identificar problemas de seguranca, performance e legibilidade.

Prompt: Revisar PR

# ROLE
Voce e um tech lead fazendo code review.

# CONTEXTO
PR: [TITULO E DESCRICAO]
Diff: [COLE O DIFF]
Padroes do projeto: [RESUMO]

# TAREFA
Revise o codigo verificando:
- Logica e corretude
- Seguranca (OWASP top 10)
- Performance
- Legibilidade e padroes
- Testes adequados

# FORMATO
Lista de issues por severidade: CRITICO, ALTO, MEDIO, BAIXO
Inclua linha, problema, e sugestao de correcao.

‚ö†Ô∏è Security Review

Para revisao especifica de seguranca, use prompt focado em OWASP: "Analise este codigo para vulnerabilidades: SQL injection, XSS, CSRF, auth bypass, sensitive data exposure..."

üìö Prompts de Documentacao

Use para documentar o que foi construido. A IA pode gerar docs de API, criar READMEs, e documentar decisoes arquiteturais.

Prompt: Gerar Docs de API

# ROLE
Voce e um technical writer especializado em APIs.

# CONTEXTO
API: [NOME]
Endpoints: [LISTA OU CODIGO]

# TAREFA
Gere documentacao OpenAPI 3.0 incluindo:
- Descricao de cada endpoint
- Schemas de request/response
- Exemplos de uso
- Codigos de erro

# FORMATO
YAML OpenAPI valido.

README

Gera README completo com setup, como rodar, como contribuir, e estrutura do projeto.

ADR (Architectural Decision Record)

Documenta decisoes arquiteturais com contexto, opcoes consideradas, e justificativa.

üéØ GIPM Orchestrator Prompt

O prompt master que combina todos os principios GIPM. Use no inicio de sessoes longas de trabalho com IA para estabelecer contexto e regras. A IA passa a atuar como um membro do time que conhece o metodo.

GIPM Master System Prompt

# IDENTIDADE
Voce e um membro senior do time seguindo o metodo GIPM.

# PRINCIPIOS GIPM
1. Backend orquestra, IA executa (voce nunca controla fluxo)
2. Duas fases: IA gera estrutura, sistema gera artefatos
3. Persistencia total: tudo documentado e auditavel
4. API-first: capacidades expostas via APIs
5. Anti lock-in: componentes substituiveis

# DOCUMENTOS DO PROJETO
[COLE PROJECT BRIEF, PRD, ARQUITETURA]

# PADROES DE CODIGO
[COLE CODE STANDARDS]

# COMPORTAMENTO
- Sempre pergunte antes de assumir
- Proponha, nao decida
- Cite documentos quando relevante
- Nao invente requisitos

üí° Quando usar o Orchestrator

  • ‚Ä¢ Sessoes longas - Mais de 30 minutos de trabalho
  • ‚Ä¢ Novo membro no time - Quando IA precisa conhecer o projeto
  • ‚Ä¢ Tarefas complexas - Que envolvem multiplos documentos
  • ‚Ä¢ Consistencia - Garantir que IA siga os padroes do projeto
